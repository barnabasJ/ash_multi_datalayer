1. **HIGH — H1 memoization acceptance can pass while header-scoped auth remains broken.** `docs/tasks/20260707-second-review-fixes/h1-remote-calc-fetch-unauthenticated.md:46-53` correctly notes the bundle memo key excludes both actor and headers, but its required repro only varies actors. That lets an implementation key the process memo by actor while still reusing values across different caller-supplied `context[:ash_remote][:headers]`, even though those headers are explicitly in scope at `h1-remote-calc-fetch-unauthenticated.md:39-44`. Current source confirms the bundle key is only `{pk_values, specs, context.tenant}` at `../ash_remote/lib/ash_remote/remote_calculation.ex:68-84`, and explicit headers are part of the request auth context at `../ash_remote/lib/ash_remote/data_layer.ex:414-423`. Add a same-actor/different-explicit-headers memoization repro or require clearing/scoping the memo per load.

2. **HIGH — L8 omits `/rpc/validate`, so authorization can still be skipped on the validation endpoint.** `docs/tasks/20260707-second-review-fixes/l8-rpc-authorize-flag.md:31-40` requires read/create/update/destroy/fetch-helper coverage, but not `validate_action`. The repo exposes `/rpc/validate` at `../ash_remote/lib/ash_remote/server/router.ex:53-60`, and `validate_action/3` still builds `Ash.Query`/`Ash.Changeset` subjects via `subject_opts` without `authorize?: true` at `../ash_remote/lib/ash_remote/server.ex:309-330` and `:415-418`. The public docs also describe `Server.run_action` and `validate_action` together as applying Ash authorization posture (`../ash_remote/lib/ash_remote/rpc.ex:52-55`). The task can be closed while `authorize :when_requested` policies are still bypassed for RPC validation unless this endpoint is tested or explicitly excluded with a reason.

3. **MEDIUM — B7’s expected synced-entry contract is internally inconsistent and under-tests `discard/1`.** The fix text allows a synced stale handle to be either “idempotent no-op success (or typed error)” (`docs/tasks/20260707-second-review-fixes/b7-resolution-verb-synced-guard.md:39-43`), while the checklist requires “no-op” and also says non-parked entries are rejected with typed errors (`b7-resolution-verb-synced-guard.md:47-50`). Pick one final contract. Also, the repro says unfixed code fails “by re-pushing” for each verb, but current `discard/1` does not re-push: non-create discard destroys the entry and kicks next at `lib/ash_multi_datalayer/orchestrator/local_outbox/api.ex:130-146`, while create discard re-reads `record_chain/1`, which excludes synced rows at `api.ex:518-526`. A test that only watches target pushes can pass while `discard/1` still mutates/deletes synced entries.
