## Findings

1. **HIGH - PRE can be marked DONE while the commit-fragility it exists to remove is still open.** `docs/tasks/20260707-second-review-fixes/00-index.md:69-72` says PRE is DONE if the per-repo commits are made **or** a skip decision is recorded, and `pre-checkpoint-commit.md:43-51` repeats that either repo may be skipped while still concluding retained-regression tasks can rely on the fixed-in-tree items being durable. But the stated defect is that Cat A fixes only exist in uncommitted working trees (`00-index.md:80-91`, `pre-checkpoint-commit.md:12-20`), and a skip decision does not make them durable. This lets PRE, FINAL (`final-gates.md:64`), A0/R0 retained-regression work, and B1/L12 Cat-A regressions all close while the underlying branch-switch/backslide hazard remains. If skip is allowed, the tracker needs an explicit alternate closure path that reclassifies Cat A items and removes downstream durability assumptions, not just `DONE`.

2. **MEDIUM - The canonical-tenant requirement conflates three different values and can be satisfied by breaking target calls or persisted nil-tenant outbox rows.** The index requires one canonical string key for read coverage, write invalidation, outbox filters, notifications, and target calls (`docs/tasks/20260707-second-review-fixes/00-index.md:124-141`), and B3 requires all five call sites, including target calls, to return the identical canonical key (`b3-tenant-from-filter-dead-code.md:97-103`). Current source has distinct semantics: coverage partitions map `nil` to `:__global__` (`lib/ash_multi_datalayer/coverage.ex:21-24`), LocalOutbox stores no-tenant entries as SQL `NULL` via `stringify(nil)` (`lib/ash_multi_datalayer/orchestrator/local_outbox/write.ex:273,310`) and filters `nil` as `IS NULL` (`lib/ash_multi_datalayer/orchestrator/local_outbox/api.ex:565`), while target calls pass the actual Ash tenant value to the target layer (`lib/ash_multi_datalayer/orchestrator/local_outbox/flush.ex:166-178`, `api.ex:196-200`). A single "never nil" sentinel cannot mean both "all tenant partitions" and "the tenant argument to an Ash target call", and replacing persisted/target `nil` with a sentinel would change behavior for unscoped rows. Split the acceptance criteria into canonical partition key, all-partitions scope sentinel, and target-call tenant value.

3. **LOW - H2 still contains a false/stale source claim about `accepted_keys/1`.** `h2-non-pk-upsert-identity-accept-truncation.md:25-31` correctly says `accepted_keys/1` falls back to all attributes including `public? false`, matching `../ash_remote/lib/ash_remote/data_layer.ex:367-371`. But the Fix section then says it "falls back to all public attrs otherwise" at `h2-non-pk-upsert-identity-accept-truncation.md:48-50`. That contradicts both the same task and the current source, and can mislead the implementation/testing scope for the accept-list half. Change the stale wording to "all attributes" and keep the explicit-`accept` scoping.
