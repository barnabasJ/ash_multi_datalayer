## Findings

1. `docs/tasks/20260707-second-review-fixes/m11-decoder-crashes-nil-single-object.md:40-63` does not require rejecting a bare object for ordinary non-`get?` list reads. Current source confirms `Protocol.parse_run/1` accepts any present `"data"` value (`../ash_remote/lib/ash_remote/protocol.ex:62`), `DataLayer.do_run_query/2` passes it straight to `Decoder.decode_records/3` (`../ash_remote/lib/ash_remote/data_layer.ex:130-132`), and `Decoder.decode_records/3` currently only handles `%{"results" => ...}` or lists (`../ash_remote/lib/ash_remote/decoder.ex:26-32`). The task requires a `get?` bare-object hit to decode as `[record]` and a non-`get?` `data: null` to error, but a fix could add `decode_records(map) -> [decode_record(...)]` globally and satisfy all listed checks while silently accepting malformed bare-object responses for normal list reads. Add an acceptance check that non-`get?` reads with bare-object `data` return a typed protocol error.

2. `docs/tasks/20260707-second-review-fixes/m1-upsert-skipped-badmaperror.md:10-12` points implementers to `lib/ash_multi_datalayer/orchestrator/local_outbox/write.ex:213-240` “via `Snapshot.record_pk`”, but the actual crash site is outside that range: `enqueue_entries/7` calls `Snapshot.record_pk(resource, record)` at `write.ex:267-270`, and the related tenant extraction is at `write.ex:273`. The cited range only shows `async_run/3` forwarding `record` into `enqueue_entries/7`; it excludes the load-bearing `Map.get` crash path. This is a misleading source ref because a fix aimed at the cited range could change `async_run/3` flow while missing the actual snapshot/tenant extraction defects.

3. `docs/tasks/20260707-second-review-fixes/final-gates.md:65-73` lets FINAL check PRE as complete if “its skip decision is recorded”. That conflicts with `pre-checkpoint-commit.md:43-65` and `00-index.md:69-73`, which require any PRE skip to include the durability plan because a bare skip does not make Cat A fixed-in-tree items durable. As written, FINAL can be accepted with only a recorded skip while retained-regression Cat A work remains commit-fragile. Require the same “skip recorded with durability plan” condition here.

4. `docs/tasks/20260707-second-review-fixes/00-index.md:280-283` says the retained-regression inventory includes Cat C docs-only items and then instructs “write each repro, expect it to PASS”. This contradicts the same index’s Cat C/docs-only language (`00-index.md:89-90`, `00-index.md:118-121`) and the harness tasks (`a0-mdl-repro-harness.md:45-49`, `r0-ash-remote-repro-harness.md:44-47`), which explicitly say docs-only confirmations have no repro. This creates an impossible global gate for Cat C items such as L13 item 4, L10 doc items, P5 non-runtime parts, and L9’s documentation checkbox. Limit the “write each repro” sentence to Cat A/B behavior regressions.
