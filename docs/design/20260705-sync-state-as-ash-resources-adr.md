# 20260705-Sync-State-As-Ash-Resources-ADR

**Status**: Proposed **Date**: 2026-07-05 **Deciders**: Barnabas Jovanovics

## Decision Drivers

- The [LocalOutbox RFC](./20260705-local-outbox-orchestrator-rfc.md)'s first
  draft specified a bespoke drainer GenServer, a bespoke `Outbox.Store`
  behaviour with an ETS implementation, bespoke retry/backoff scheduling, and a
  bespoke dead-letter set with its own query API. Every one of those is a
  re-implementation of something that already exists in the ecosystem we are
  building for: durable queues with retries and backoff are Oban's whole job,
  and "typed state you can query, subscribe to, and attach hooks to" is an Ash
  resource's whole job.
- Sync failures are *application* events, not library internals. A local-first
  UI must render "3 changes not synced — retry?"; an app must react to a
  conflict with domain logic. If parked entries are rows of an Ash resource the
  application owns, the app hooks in with the tools it already knows —
  notifiers, pub_sub, changes on actions, plain queries — instead of learning a
  library-specific callback registry.
- Pausing/resuming replication (device offline/online) is a first-class
  requirement. Oban queues already pause and resume at runtime
  (`Oban.pause_queue/2`); a bespoke drainer would grow the same feature by
  hand.
- The RFC draft's hardest open problem was the **durability invariant**: a
  durable local layer paired with a volatile outbox silently diverges after a
  crash. If the outbox is an Ash resource on the *same repo* as the local
  layer (ash_sqlite on the client device), the local write and the enqueue can
  commit in **one transaction** — the invariant is satisfied by construction,
  not by documentation.
- The client-device stack lines up end to end: ash_sqlite for the local layer,
  the same SQLite database for the outbox resource, and Oban's `Lite` engine
  running on that database. One file on disk holds the data, the queue, and
  the job state.
- ProvenCoverage's coverage ledger is currently a hand-rolled ETS table
  (`Coverage` + `TableOwner`). Modeling it as an ETS-backed Ash resource makes
  it inspectable and hookable with the same tools, and proves the "orchestrator
  state is resources" pattern on both strategies.
- Verified against the local hex cache (this sandbox is offline): oban 2.23.0
  (`Oban.Engines.Lite` with unique-job support), ash_oban 0.8.10 (triggers
  with `where`/`sort`, `on_error`, `max_attempts`, per-trigger `queue`,
  `scheduler_cron: false`, `AshOban.run_trigger/3`; capability-checked against
  the data layer, not Postgres-specific), ash_sqlite 0.2.17, ecto_sqlite3,
  exqlite, igniter 0.8.2.

## Context

The [orchestrator behaviour ADR](./20260705-orchestrator-behaviour-adr.md)
extracts routing *policy* into strategy modules. Each strategy carries state:
ProvenCoverage has the coverage ledger; LocalOutbox has the outbox and its
dead-letter set. The question this ADR answers is **how that state is stored,
executed against, and exposed** — as private library machinery, or as
first-class Ash resources in the host application with the async work running
on the host's job runner.

## Decision

**Orchestrator sync state is modeled as Ash resources owned by the host
application, generated by igniter installers, and declared to the orchestrator
via its opts. Asynchronous replication work is executed by Oban through
ash_oban triggers on those resources — the library ships no queue, no drainer,
and no scheduler of its own.\* Transient failures are Oban's to retry; terminal
failures are state transitions on the resource that the application can hook.**

\* One named exception (plan-review pass-4 P2): ash_oban 0.8.10 hardcodes the
default `Oban` instance on every enqueue path, so supporting named instances
requires MDL to ship a **thin sweep worker** ("query the outbox resource for
pending entries + insert jobs into the configured instance") and to own the
job-insertion call sites (`scheduler_cron false` on the trigger). ash_oban's
worker/trigger/action machinery still executes everything; the "no scheduler"
claim holds unqualified only for single-default-instance deployments. See the
plan's Phase 3 mechanism.

Concretely, four sub-decisions:

### 1. State resources are generated into the app, shaped by library extensions

The library ships Spark DSL extensions that inject the required shape
(attributes, actions, identities, the ash_oban trigger), and igniter
generators that create the resource modules in the user's application:

```elixir
# generated by: mix ash_multi_datalayer.gen.outbox TodoClient.Sync.OutboxEntry \
#                 --repo TodoClient.Repo --queue todo_sync
defmodule TodoClient.Sync.OutboxEntry do
  use Ash.Resource,
    domain: TodoClient.Sync,
    data_layer: AshSqlite.DataLayer,
    extensions: [AshMultiDatalayer.Sync.OutboxEntry, AshOban]

  sqlite do
    table "amd_outbox_entries"
    repo TodoClient.Repo
  end

  outbox_entry do
    queue :todo_sync            # the Oban queue flush jobs run on
    max_attempts 10             # transient-retry budget before parking
  end
end
```

The extension owns the contract (so the orchestrator can rely on the shape);
the app owns the module (so the user can attach notifiers, pub_sub, policies,
and extra actions without any library hook API). The orchestrator finds it
through its opts: `orchestrator {LocalOutbox, outbox_resource: TodoClient.Sync.OutboxEntry, ...}`,
enforced by `validate_opts/2`.

The same pattern applies to ProvenCoverage's ledger: an
`AshMultiDatalayer.Coverage.LedgerEntry` extension over an ETS-backed
resource, `ledger_resource` opt, generator for users who want to customise it,
and a library-shipped default resource so zero-config keeps working.

### 2. Oban executes the async work, via ash_oban triggers

The generated outbox resource carries an ash_oban trigger:

- **Immediate path**: after a local write commits, the orchestrator inserts
  a flush job for each entry row directly into the configured Oban instance
  (MDL-owned insertion against ash_oban's generated worker — ash_oban
  0.8.10's own enqueue paths hardcode the default `Oban` instance, so MDL
  bypasses exactly its insertion calls; plan-review-2 R1) — the flush job
  is enqueued right away, no polling latency.
- **Sweeper path**: an MDL-owned sweep worker in the instance's cron
  (`where: state == :pending`, `sort: [seq: :asc]`;
  `scheduler_cron false` on the trigger) re-enqueues anything missed — crashed jobs, entries
  enqueued while Oban was down, chains unblocked by a resolution. The classic
  transactional-outbox pattern: rows are the truth, jobs are ephemeral
  pointers to rows.
- **Retries**: a transient flush failure errors the job; Oban retries with its
  backoff up to `max_attempts`, visible in the standard Oban tooling.
- **Terminal failures**: after the last attempt, ash_oban's `on_error` invokes
  the resource's `:park` update action. Semantic rejections and conflicts park
  directly from within the flush action (retrying cannot help them) without
  burning the retry budget.
- **Pause/resume**: `LocalOutbox.pause_sync/1` / `resume_sync/1` wrap
  `Oban.pause_queue/2` / `Oban.resume_queue/2` on the entry resource's queue.
  Offline detection stays the application's job; the library provides the
  lever. (Queue pause state is runtime-only — an app that boots offline
  re-pauses on startup; documented, and the example demonstrates it.)

The host application owns the Oban instance (repo, engine, queue
concurrency); the library never starts Oban. On a client device that is
`Oban.Engines.Lite` on the same SQLite file as the local layer; on a server
deployment it is the existing Postgres-backed Oban.

**Consequence**: LocalOutbox requires a SQL-backed outbox resource
(ash_sqlite or ash_postgres) because Oban requires a SQL engine. The RFC
draft's ETS outbox store is **dropped** — an in-memory outbox with no durable
queue behind it was the configuration the durability invariant warned about
anyway. (ProvenCoverage's ledger keeps ETS: it is derived, loss-tolerant
cache metadata and involves no async work.)

### 3. Errors are modeled on the resource, in two classes

- **Transient** (connection refused, timeout, HTTP 5xx from the ash_remote
  backend): the domain of the job runner. The entry stays `:pending`; the
  Oban job carries the attempt count and backoff. The library classifies
  errors conservatively — anything not provably semantic is transient.
- **Terminal** (`:rejected` — constraint/validation/authorization error from
  the replica; `:conflict` — stale-check mismatch, someone else wrote the
  replica): a state transition `pending → parked` on the entry, recording
  `error_class`, the replica's error verbatim, and (for conflicts) the
  replica's current row. Parking fires the resource's own notifiers — the
  application reacts by subscribing to the resource it owns (pub_sub on
  `:park`, a change on the action, or polling `parked` reads). Resolution
  verbs (`retry`, `discard`, `discard_local`, `force`, `rebase`) are actions
  and orchestrator API over the same rows, per the RFC.

No separate "SyncFailure" resource: the parked entry *is* the failure record,
with the chain semantics (a parked head blocks its PK chain) attached to the
same row. A projection resource can always be layered on by the app.

### 4. Bootstrap via igniter

`mix ash_multi_datalayer.install` (igniter installer): supervisor into the
app tree, formatter/config wiring. `mix ash_multi_datalayer.gen.outbox`:
generates the sync domain + outbox entry resource, composes the existing
`ash_oban.install` and ash_sqlite setup (repo, config, migrations) where
missing, and prints the orchestrator opts to paste. A matching
`...gen.ledger` exists for the ProvenCoverage ledger resource. Generators are
additive and idempotent (igniter semantics), and the example app is generated
with them — the generator is tested by using it.

## Consequences

### Positive

1. Retries, backoff, scheduling, concurrency, pausing, and job observability
   are Oban's — battle-tested and already operated by the target audience —
   instead of a new bespoke drainer.
2. Terminal sync failures are queryable, subscribable domain state; the app
   hooks them with standard Ash tools on modules it owns.
3. Local write + outbox enqueue commit atomically when they share a repo —
   the durability invariant becomes a construction, not a warning.
4. The client stack is one SQLite file: local data, outbox, job queue.
5. The ledger-as-resource port proves the pattern generalises across
   strategies and makes `mix ash_multi_datalayer.inspect` mostly a query.

### Negative

1. LocalOutbox grows hard dependencies: oban + ash_oban + a SQL data layer
   for the outbox (all optional deps of the library, required only by this
   strategy — enforced by `validate_opts/2` with clear errors).
2. Flush ordering is not free: Oban does not guarantee FIFO even at
   concurrency 1 (retry backoff reorders). Per-PK ordering must be enforced
   by the flush action itself (chain-head check against `seq`), not assumed
   from the queue. This is a correctness-critical piece of library code and
   is specified in the RFC.
3. Ash-action overhead on hot paths. For the outbox this is amortised by
   network I/O; for the ProvenCoverage ledger the probe path
   (`covers?`/`record` on every read) currently costs an ETS select, and an
   Ash read is measurably more. The ledger port is benchmark-gated and keeps
   the storage seam so a raw-ETS fast path can return if it regresses
   (see the plan's Phase 5 gate).
4. Generated resources can drift from the extension's expectations across
   library upgrades; the extension verifier re-checks the contract at compile
   time, and upgraders ship as igniter patches.

## Alternatives Considered

### Alternative 1: Bespoke drainer + `Outbox.Store` behaviour (the RFC's first draft)

- Good, because no new dependencies; works on pure-ETS clients with no repo.
- Bad, because it re-implements retries, backoff, scheduling, pausing,
  dead-lettering, and introspection that Oban ships; every feature request
  (rate limiting, cron sweeps, observability) re-opens it.
- Bad, because the ETS store it defaults to is exactly the volatile-outbox
  configuration the durability invariant warns against.

**Why not**: strictly more code for strictly less capability; the one thing
it enables (repo-less clients) contradicts the durability story local-first
needs.

### Alternative 2: Plain Oban workers, no ash_oban, outbox state in Oban job args

The write path inserts Oban jobs directly; no outbox resource at all — the
job table *is* the outbox.

- Good, because fewest moving parts.
- Bad, because the sync state stops being domain state: no Ash queries, no
  notifiers, no policies, no user hooks without raw SQL against `oban_jobs`;
  parked failures live as `discarded` jobs with the error buried in a JSON
  column; per-PK chains require peeking into other jobs' args.

**Why not**: it optimises away exactly the property the user experience
needs — sync state as first-class, reactable application data.

### Alternative 3: Separate dead-letter resource for terminal errors

Parked entries copied into a `SyncFailure` resource; outbox stays
short-lived.

- Good, because the failure log can outlive entry cleanup.
- Bad, because chain-blocking needs the parked row *in* the outbox ordering
  anyway; two resources mean two sources of truth for "is this record
  synced?".

**Why not**: the entry's state machine already expresses it; an app that
wants an audit log can subscribe to `:park` and write its own.

### Alternative 4: Library-internal resources (not generated into the app)

Ship `AshMultiDatalayer.Sync.DefaultOutboxEntry` and use it implicitly.

- Good, because zero setup.
- Bad, because the user can't attach notifiers/policies/actions to a module
  they don't own; multi-repo apps can't point it at the right repo; and the
  resource needs a domain, which the library shouldn't own.

**Why not** as the *only* mode: kept for the ProvenCoverage ledger default
(where zero-config matters and nobody customises); rejected for the outbox,
where hooking in is the point. The generator makes the setup cost one
command.

## Validation

- The LocalOutbox test suite drives flush, retry, park, and resolution purely
  through the generated resource + a real Oban instance (Lite engine, `:test`
  mode inline/manual) — proving no hidden library state exists outside the
  resources and the job table.
- The example app's offline toggle pauses the queue, accumulates entries,
  resumes, and drains — observable in the outbox resource's rows.
- Benchmark gate on the ProvenCoverage ledger port (plan Phase 5): read-probe
  overhead within the agreed budget or the raw-ETS fast path stays.
