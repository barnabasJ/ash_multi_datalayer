# RFC: `Orchestrator.LocalOutbox` — local-authoritative layers with asynchronous replication

**Status**: Draft v2 — for review **Date**: 2026-07-05
**Author**: Barnabas Jovanovics
**Depends on**: [Orchestrator behaviour ADR](./20260705-orchestrator-behaviour-adr.md),
[Sync-state-as-Ash-resources ADR](./20260705-sync-state-as-ash-resources-adr.md)

> **v2 note**: the first draft specified a bespoke drainer GenServer, an
> internal `Outbox.Store` behaviour with an ETS implementation, and a
> four-callback `Handler` behaviour. All three are replaced: the outbox is an
> **Ash resource in the host application** (generated by igniter), the async
> work runs on **Oban via ash_oban triggers**, and applications react to
> failures through the resource's own notifiers + a resolution API. Rationale
> in the [sync-state ADR](./20260705-sync-state-as-ash-resources-adr.md).

## Problem

Some compositions want the *local* layer to own the data: reads and writes are
served entirely from a fast node-local layer, and a slower/remote layer (a
Postgres instance, an `ash_remote` backend) is a store the local state is
**synced to after the fact**. The existing strategy (ProvenCoverage) has the
opposite authority: the last layer is the source of truth and the local layer
is a proven cache. Inverting authority changes the read path, the write path,
`can?`/transaction semantics, and — the genuinely hard part — **what happens
when the asynchronous replication fails or conflicts, long after the caller's
write returned `{:ok, record}`**.

The motivating use case is a local-first client: an ash_sqlite layer on the
device is the working store, an ash_remote layer syncs to the backend, sync
pauses while offline and resumes on reconnect, sync failures are visible,
reactable application state — and changes made by *other* clients arrive as
ash_remote realtime notifications that must land in the local store (see
Inbound changes below).

This RFC specifies `AshMultiDatalayer.Orchestrator.LocalOutbox`. Per the
naming convention in the behaviour ADR, the module is named for its mechanism
(an ordered outbox draining to the replica), not the "local-first" category.

## Relationship to the no-write-behind ADR

The [2026-04-17 ADR](./20260417-no-write-behind-in-v1-adr.md) cut `:write_behind`
from ProvenCoverage's topology, and constrained any future async design:
*layers earlier in `read_order` than the authoritative layer must be updated
synchronously*. LocalOutbox **refines rather than contradicts** that decision:

- The authoritative layer *is* the first (and only) layer in `read_order`, so
  the invariant is satisfied trivially — asynchrony applies only to a layer
  that reads never consult.
- The ADR's first hard problem — a coverage-missing read falling through to a
  not-yet-flushed layer and reading stale from below — is eliminated
  structurally: LocalOutbox reads **never fall through**. The local layer is a
  complete copy (see Hydration), not a partial cache, so there is no ledger
  and no miss path.
- The ADR's second hard problem — durability of unflushed writes across a
  crash — is *solved by construction* in the recommended configuration: the
  outbox is a resource on the same repo as the local layer, and the local
  write + enqueue commit in one transaction (see Durability).

## Dependencies

This strategy requires what ProvenCoverage does not:

- **oban** (host-owned instance) and **ash_oban** (trigger machinery) —
  optional deps of the library, validated present by `validate_opts/2`.
- A **SQL-backed outbox resource** (ash_sqlite or ash_postgres): Oban needs a
  SQL engine, and a durable outbox is the point. On a client device the
  intended stack is one SQLite file carrying the local layer's tables, the
  outbox table, and `oban_jobs` (engine: `Oban.Engines.Lite`). All packages
  verified available in this sandbox's hex cache (oban 2.23.0, ash_oban
  0.8.10, ash_sqlite 0.2.17, ecto_sqlite3, exqlite).

## Topology and DSL

```elixir
multi_data_layer do
  orchestrator {AshMultiDatalayer.Orchestrator.LocalOutbox,
    outbox_resource: TodoClient.Sync.OutboxEntry,  # generated; see below
    conflict_detection: :off,                     # :off (default) | {:stale_check, :updated_at}
    hydrate: :if_empty                            # :if_empty (default) | :on_start | :manual
  }

  layer :local, AshSqlite.DataLayer
  layer :remote, AshRemote.DataLayer

  read_order [:local]              # verifier-enforced: exactly the local layer
  write_order [:local, :remote]    # first = synchronous authority, rest = outbox targets
end
```

The existing `read_order`/`write_order` lists keep their shape; LocalOutbox
*interprets* them: `hd(write_order)` is the synchronous authoritative layer
(also the `authority/1` and `transaction_layer/1` answer), every subsequent
layer is an asynchronous replication target. `validate_opts/2` + strategy
verifiers enforce `read_order == [local]`, `hd(write_order) == local`, at
least one replication target, the outbox resource implementing the
`AshMultiDatalayer.Sync.OutboxEntry` extension, and oban/ash_oban loaded.

`can?` consequences of the inverted authority: **everything read-shaped
delegates wholesale to the local layer** — reads never touch another layer,
and the local copy is complete, so aggregates, `aggregate_filter`,
`distinct`, joins, `:select`, `source/1`, and `:transact` are all the local
layer's own answers, not inherited hardcodes or intersections (the replica
cannot participate in a caller-visible operation, so its capabilities must
not constrain — or advertise — anything). Write features are the local
layer's, plus a verify-time check that each target accepts PK-upsert/destroy
(all propagation ever sends). The only hardcoded `false`s are the
orchestration-bypass guards — `update_query`/`destroy_query`/`{:atomic,_}`/
bulk would mutate the local layer without enqueueing outbox entries (silent
divergence), and stay refused even where the local layer supports them
(ash_sqlite answers `true` to `update_query`). Each guard carries its reason
in a comment; the general derivation principle lives in the behaviour ADR.

## The outbox resource

Generated into the application by
`mix ash_multi_datalayer.gen.outbox TodoClient.Sync.OutboxEntry --repo TodoClient.Repo --queue todo_sync`
(see the sync-state ADR for the ownership rationale). The
`AshMultiDatalayer.Sync.OutboxEntry` extension injects the contract:

**Attributes** — `id` (uuid), `seq` (monotonic per outbox: autoincrement
integer on SQL layers — the per-PK ordering key), `write_ref` (uuid shared by
all entries of one write, the unit `await/2` waits on), `resource` (string),
`tenant`, `record_pk` (map), `op` (`:create | :update | :upsert | :destroy`),
`payload` (the record the local layer *returned* — never a re-run of the
changeset, same rationale as FR3.6), `base_image` (`changeset.data` for
updates/destroys, `nil` for creates; kept for conflict detection), `target`
(layer name), `state` (`:pending | :parked`), `error_class`
(`nil | :transient_exhausted | :rejected | :conflict`), `last_error` (map),
`remote_snapshot` (the replica's row at conflict time, `nil` otherwise),
`parked_at`, timestamps.

**Actions** — `enqueue` (create), `flush` (the worker action — see below),
`park`, `retry` (parked → pending + re-trigger), `discard`, `pending` /
`parked` / `for_record` reads.

**ash_oban trigger** — injected by the extension:

```elixir
trigger :flush do
  action :flush
  queue :todo_sync                 # from the outbox_entry DSL
  where expr(state == :pending)
  sort [seq: :asc]
  max_attempts 10                  # transient-retry budget
  on_error :park                   # exhausted transients dead-letter
  scheduler_cron false             # sweeping is MDL-owned; see below
  worker_module_name ...           # generated; MDL inserts against it
end
```

**Oban-instance note** (plan-review-2 R1 + pass-4 P1): ash_oban 0.8.10
calls bare `Oban.insert!/1` on every enqueue path — it cannot target a
named instance. MDL therefore **owns its Oban touchpoints through one
enqueue helper with exactly four call sites**: the immediate kick (write
path), the sweeper (an MDL-owned sweep worker registered in the instance's
cron plugin — `scheduler_cron false` above disables ash_oban's own
scheduler), the **chain continuation** in the flush success path, and the
**retry re-trigger** — the last two are library-internal and must never
fall back to `AshOban.run_trigger/3` (which would land in the default
instance and stall the chain). Worker-side sites resolve the instance from
`job.conf.name`; caller-side sites from the same process-scoped
establishment as the dynamic repo. ash_oban still provides the
worker/trigger/action machinery — only its insertion calls are bypassed.
The `oban_instance` option (default `Oban`) lives on the `outbox_entry`
section and the orchestrator opts.

Because the module lives in the app, the app attaches its own notifiers,
pub_sub, policies, or extra actions directly — that *is* the error-hook API.

## Read path

All reads run on the local layer, unconditionally. No coverage ledger, no
subsumption proof, no backfill. Aggregates and calculations evaluate on the
local layer (it holds the complete data set); the SQL-passthrough and fold
machinery of ProvenCoverage does not apply.

**Hydration.** The local layer must be a complete copy for this to be sound.
v1 of the strategy supports *full hydration only*:

- `hydrate: :if_empty | :on_start | :manual`. `:if_empty` (default, the right
  mode for a persistent SQLite local layer) streams every record of the
  resource (per tenant) from the last `write_order` layer only when the local
  layer holds no rows; `:on_start` always re-streams (upsert semantics,
  idempotent); both run under the strategy's `child_specs/1` before the
  resource is served. `LocalOutbox.hydrate(resource, tenant \\ nil)` for
  `:manual` / re-hydration.
- Hydration never runs while the outbox holds pending or parked entries for
  the resource — pulling the replica over unflushed local writes would erase
  them. `hydrate/2` returns `{:error, :outbox_not_empty}`; the operator
  resolves or discards first.
- Partial hydration (local layer holds a subset, misses fall through) is
  explicitly out of scope: it reintroduces the coverage ledger and converges
  back to ProvenCoverage. If that hybrid is ever wanted, it is a third
  strategy, not an option here.

## Write path

1. **Synchronous authoritative write** to the local layer — fail-fast, error
   returned verbatim, nothing else touched (same contract as ProvenCoverage's
   step 1).
2. **Enqueue** one outbox entry per replication target via the outbox
   resource's `enqueue` action, sharing a `write_ref`. When the local layer
   and the outbox resource share an **Ecto repo** (the SQLite client stack),
   steps 1–2 run in **one transaction** — the write and its replication
   intent commit or fail together. The predicate is repo identity wrapped in
   a raw `Repo.transaction/1`, *not* `can?(:transact)`: ash_sqlite hardcodes
   that capability `false` even though SQLite transactions work fine through
   the repo (plan-review F1). Internal atomicity needs the repo, not the Ash
   data-layer capability — caller-visible `can?(:transact)` keeps the local
   layer's honest answer.
3. **Return** `{:ok, record}` with `record.__metadata__.outbox_ref` set to the
   `write_ref`.
4. **Kick the drain**: insert a flush job for each enqueued entry into the
   configured `oban_instance` (MDL-owned insertion against the generated
   worker — see the Oban-instance note above), *after* commit; if the kick
   is lost to a crash, the MDL sweep worker picks the row up on the next
   cron tick — rows are the truth, jobs are ephemeral pointers.

## Flush execution and ordering

The `:flush` action on an entry, executed inside the Oban worker:

1. **Chain-head check.** Query for a pending-or-parked entry with the same
   `(resource, record_pk, target)` and lower `seq`. If one exists, this entry
   is not the chain head: complete the job without flushing (the head's
   completion re-triggers; the sweeper is the backstop). Per-PK FIFO is
   enforced *here*, in data, because Oban does not guarantee execution order
   (retry backoff reorders even at concurrency 1).
2. **Flush.** Apply the entry to `target` as a PK-upsert (create/update/
   upsert) or PK-destroy through the same `Backfill` helpers ProvenCoverage's
   propagation uses, honouring `conflict_detection` (below).
3. **Outcome.**
   - Success → delete the entry (a synced entry has no further meaning),
     then insert a flush job for the next pending entry of the same chain
     (MDL-owned insertion), if any.
   - **Offline-class error** (connection refused, host unreachable, DNS
     failure — the target is not *reachable*, as opposed to reachable but
     misbehaving) → `{:snooze, interval}`. Verified against the Oban 2.23
     engine source (Basic, delegated unchanged by Lite): `snooze_job`
     re-schedules the job **and increments `max_attempts`**, so snoozing
     costs zero retry budget. A device that goes offline without anyone
     calling `pause_sync/1` therefore retries gently forever and parks
     nothing — being paused is a battery/UX nicety, never a correctness
     requirement.
   - **Transient error** (timeout, HTTP 5xx, transport failure while
     connected — the conservative default class for anything not provably
     offline or semantic) → raise; Oban retries with backoff; after
     `max_attempts`, `on_error` runs `park` with
     `error_class: :transient_exhausted`.
   - **Rejection** (the replica rejected the write semantically: constraint
     violation, authorization, replica-side validation) → `park` immediately
     with `error_class: :rejected` and the replica's error verbatim; the job
     completes (retrying cannot help; the retry budget is for transients).
   - **Conflict** (stale-check mismatch, only with detection on) → fetch the
     replica's current row into `remote_snapshot`, `park` with
     `error_class: :conflict`; job completes.

### Chain-blocking

Per-PK ordering is a correctness requirement, not an optimisation: an update
must never reach the replica before its create, and a destroy must be last.
When an entry parks, *its PK chain blocks*: later entries for that PK fail the
chain-head check and hold, while other PKs continue to drain. Resolving the
parked head (below) unblocks or discards the chain. Cross-PK ordering is not
guaranteed and not needed — each record converges independently.

## Durability

**Invariant: the outbox's durability must be ≥ the local layer's.** With the
resource-based outbox this is a configuration matrix, checked in docs and (for
the detectable part) `validate_opts/2`:

| local layer | outbox | verdict |
|---|---|---|
| ash_sqlite | ash_sqlite, same repo | **recommended** — co-commit (raw `Repo.transaction/1` on the shared repo — not gated on `can?(:transact)`, which ash_sqlite hardcodes false); nothing can be lost or forgotten |
| ash_sqlite / ash_postgres | SQL, different repo | allowed — no co-commit; a crash between steps 1 and 2 loses the *replication intent* while keeping the local write; documented, telemetry-visible (divergence detectable by comparing layers) |
| ETS | SQL outbox | allowed — a crash loses local state but the outbox survives; hydration re-pulls and pending flushes replay as idempotent upserts |
| anything durable | in-memory outbox | **rejected** — this was the RFC v1 danger case; structurally impossible now (the outbox resource must be SQL-backed for Oban regardless) |

## Sync control: pause and resume

The offline/online lever, wrapping Oban queue control on the entry resource's
queue:

```elixir
LocalOutbox.pause_sync(TodoClient.Todo)    # Oban.pause_queue(queue: :todo_sync)
LocalOutbox.resume_sync(TodoClient.Todo)   # Oban.resume_queue(queue: :todo_sync) + sweep
LocalOutbox.sync_paused?(TodoClient.Todo)
```

- While paused: writes keep committing locally and enqueueing entries; jobs
  accumulate; nothing new touches the network (a flush already *in flight*
  when the pause lands runs to completion — pause stops job fetching, it
  does not kill running jobs; if that straggler fails offline it snoozes,
  per the flush outcomes above). Exactly the offline mode a local-first
  client wants.
- Pausing is optional for correctness: an *unpaused* offline device snoozes
  offline-class failures without burning retry budget (see Flush execution),
  so nothing parks merely because nobody flipped the switch. Pause exists to
  stop the periodic connection attempts, not to protect the entries.
- Resume kicks the MDL sweep worker directly so the backlog drains without
  waiting for the next cron tick.
- Pause state is **runtime-only** (Oban semantics): an app that boots
  offline re-pauses in its own start-up (the example demonstrates the
  pattern). Connectivity *detection* is explicitly the application's job —
  the strategy provides the lever, not a network monitor.
- Multiple resources sharing one queue pause together; the generator defaults
  to one queue per sync domain, which is usually the right granularity for
  "the device went offline".

## Per-action layer targeting

Two per-call escape hatches ride on query/changeset context (the plumbing
exists — MDL's `Query` already carries context via `set_context/3`; the
exact key namespace is fixed at implementation time):

**Targeted reads — compare the states.**

```elixir
Todo
|> Ash.Query.filter(id == ^id)
|> Ash.Query.set_context(%{multi_datalayer: %{read_from: :remote}})
|> Ash.read!()
```

Routes the query raw to the named layer, for *any* strategy — the shell
short-circuits before the orchestrator is consulted, so this is one
mechanism, not per-strategy code. **Side-effect-free by contract**: no
coverage recording, no backfill, no ledger LRU touch, no outbox
interaction — a compare-read observes, never perturbs. For LocalOutbox,
`read_from: :remote` is the live replica state next to the local truth
(fresher than a parked entry's `remote_snapshot` — the conflict UI's live
diff). For ProvenCoverage, `read_from: :cache` exposes the cache layer's
*physical* rows, ghosts and all — which is the point of comparing (it is
the same observation the divergence sampler makes internally). The query
runs within the target layer's own capabilities; features it can't support
error rather than falling back. Telemetry:
`[:ash_multi_datalayer, :read, :forced]` with the layer tag, so escape-hatch
traffic is visible.

**Targeted writes — per-action write-through.** A targeted write *without*
backfill is divergence-by-API (a local-only write the outbox never hears of;
a remote-only write the local layer contradicts), so the primitive is not
"pick a layer" — it is the already-specified synchronous mode, scoped to one
action:

```elixir
changeset
|> Ash.Changeset.set_context(%{multi_datalayer: %{write_through: true}})
|> Ash.update()
```

For LocalOutbox this runs the single action under the kill-switch write
semantics: drain the record's pending PK chain inline **first** (otherwise
queued flushes would later clobber the direct write), then write the replica
synchronously, then the local layer; a replica failure fails the whole
action with nothing committed anywhere. The caller gets a hard "durable on
the server or error" guarantee for exactly the writes that need it, without
flipping the global switch — and the local layer is backfilled by
construction. For ProvenCoverage the flag is a no-op: its normal write path
is already synchronous write-through + cache backfill.

Free-form `write_to: :layer` is **rejected** in v1: with backfill it
duplicates `write_through`; without backfill it manufactures divergence.
(A "repair the replica without touching local" workflow is
`write_through` on a changeset built from the desired state, or plain
`refresh/3` afterwards if the repair happened out of band.)

## Inbound changes: refresh and the notification bridge

The outbox covers the outbound direction. The inbound direction — a change
that *never passed through this node's write path* (another client's write,
an operator fixing data on the server) — needs a way to land in the local
layer, because LocalOutbox reads never fall through: an authoritative local
copy that is never told about remote changes just stays wrong. The primitive
is a **forced read through the replica, written back into the local layer**:

```elixir
LocalOutbox.refresh(resource, pk_or_filter | :all, tenant \\ nil)
# => %{refreshed: n, deleted: n, skipped_dirty: [pk, ...]}
```

`refresh/3` reads the matching rows from the last `write_order` layer and
PK-upserts them into the local layer (via the same `Backfill` helpers);
records present locally but absent remotely within the refreshed scope are
deleted locally. It generalises the two inbound writes the design already
had: `discard_local/1` is a refresh of one parked record, and hydration is
`refresh(:all)` with an emptiness precondition.

**The dirty-chain rule (normative).** A refresh **skips any PK whose outbox
chain is non-empty** (pending or parked), reporting it in `skipped_dirty`.
Refreshing a dirty record would overwrite committed-but-unflushed local
writes, after which the still-queued flush would push the stale payload back
to the replica — strictly worse than doing nothing. A dirty record's
convergence path is the outbound one: the flush meets the remote change and
either wins (LWW under `:off`) or parks a conflict (under
`{:stale_check, _}`) — the machinery already specified above. On the
co-commit stack the dirty-check + local upsert run in one transaction on the
shared repo, so a refresh cannot interleave with a local write's commit; on
a non-transactional local layer (ETS) the window is documented and refreshes
for a resource are serialised through the strategy's process to shrink it.

**The bridge.** `../ash_remote_cache` already wires ash_remote realtime
notifications into MDL for the ProvenCoverage stack: a notifier that
invalidates coverage row-aware per notification, and a lifecycle guard that
drops the whole ledger on a websocket gap (notifications are at-most-once).
For LocalOutbox the same two events map to the inverse reactions —
invalidation is meaningless for an authoritative copy; pulling is the only
correct move:

| event | ProvenCoverage reaction | LocalOutbox reaction |
|---|---|---|
| per-record change notification | invalidate matching coverage | `refresh` that PK (dirty rule applies) |
| gap / resubscribe (writes possibly missed) | drop ledger for resource+tenant | `refresh(:all)` reconcile (upserts + deletion sweep, skipping dirty PKs) |

To keep the reaction *in the strategy* and the transport *thin*, the
orchestrator behaviour grows two optional callbacks —
`handle_external_change(resource, notification)` and
`handle_external_gap(resource, tenant)` (see the behaviour ADR) — and the
notifier/guard call those instead of hardcoding invalidation. There is **no
separate bridge package**: `ash_remote_cache` as a library dissolves — its
notifier + lifecycle guard move into **ash_remote itself** as user utilities
(an optional MDL integration; deliberately unnamed as a product), and its
example becomes the flagship mixed-strategy example (see the plan's example
phase).

The reconnect narrative composes from existing pieces: coming back online is
`resume_sync/1` (drain the outbound backlog) plus the bridge's gap event
(`refresh(:all)` pulls what was missed), in either order — dirty PKs are
skipped by the pull and converge through the push.

Payload-fastpath (deferred, recorded): change notifications usually carry
the full record, so the bridge *could* apply the payload directly (0 RPC)
instead of reading through the replica. Read-through is the v1 semantics —
it is ordering-safe and establishes truth at apply time; the fastpath is an
optimisation with its own staleness argument, taken up only if refresh RPC
volume actually hurts.

## Surfacing errors and conflicts

When the caller's write returns, only the local layer has committed; the
replica outcome arrives later, on a process the caller doesn't own. "The
caller" is three different parties at three different times:

### 1. The write's caller, at return time — an awaitable handle

Every write returns the record with `__metadata__.outbox_ref` (the
`write_ref`). A caller that needs read-your-writes *on the replica* — about to
enqueue a job that reads the replica directly, or an ash_remote client about
to navigate to a server-rendered view — makes the asynchrony explicitly
synchronous at exactly the point it matters:

```elixir
{:ok, todo} = Ash.create(changeset)

case LocalOutbox.await(todo, timeout: 5_000) do   # accepts record or ref
  :synced            -> redirect(...)
  {:parked, entries} -> surface_to_user(entries)  # entries carry error_class + last_error
  :timeout           -> proceed_optimistically(...)
end
```

`await/2` returns when every entry of the ref is flushed (`:synced`), any
parked (`{:parked, entries}`), or the timeout elapsed. Implementation: a
library notifier on the entry resource (injected by the extension) broadcasts
entry deletions/parkings to a per-node registry that `await/2` subscribes to;
the initial state is read from the resource, so no transition is missable.
It is the *only* blocking primitive; nothing else waits on the replica.

Rejected framing: making the write itself optionally synchronous
(`await: true` on the action). It duplicates `await/2`, and hides a blocking
replica round-trip inside an API whose whole contract is "local write, async
sync".

### 2. The application, asynchronously — hooks on the resource it owns

There is **no library callback behaviour** (the v1 draft's `Handler` is
dropped). Terminal failures are `park` transitions on the app's own outbox
resource; the app reacts with standard Ash machinery:

- a `pub_sub` notifier on `:park` → LiveView renders "sync failed — retry?"
- a `change` / `after_action` on `:park` → automated policy: inspect
  `error_class`, call a resolution verb (below) — e.g. auto-`force` conflicts
  on a field the app knows is last-write-wins-safe, auto-`discard_local` for
  a tombstoned record.
- plain queries over `parked` for a sync-status screen.

This is one mechanism (resource + actions) instead of two (resource + callback
registry), and the automated-policy hook runs with the full context a notifier
gets. The cost relative to v1's inline `on_conflict`: a conflict always parks
*first*, then policy reacts — the chain blocks for the reaction latency.
Accepted: resolution is rare, correctness-sensitive, and better audited as two
recorded transitions than one invisible inline verdict.

### 3. The operator / the UI — queryable outbox state + resolution verbs

Nothing terminal is only a log line; the outbox is rows:

```elixir
LocalOutbox.status(record_or_ref)   # :synced | :pending | {:parked, entries}
LocalOutbox.pending(resource, tenant)
LocalOutbox.parked(resource, tenant)

# resolution verbs, on a parked chain head:
LocalOutbox.retry(entry)          # parked → pending, re-trigger (after operator fixed the cause)
LocalOutbox.discard(entry)        # drop it; discarding a create discards its whole chain (loudly, in the return)
LocalOutbox.discard_local(entry)  # replica wins: fetch replica row, write it over the local layer
                                  # (the one sanctioned inbound write), drop the chain
LocalOutbox.force(entry)          # re-flush as a blind PK-upsert, ignoring base_image (LWW for this entry)
LocalOutbox.rebase(entry, changeset) # the app resolved the conflict into a NEW write: applied through
                                  # the full LocalOutbox write path (local commit + fresh entries
                                  # replacing the parked chain head), so the resolution itself replicates
```

The local layer **keeps** divergent local state while parked (local is
authoritative; the strategy never silently discards a user's committed
write). `status/1` is what a local-first UI polls or subscribes to for
per-record "saving… / saved / sync failed" badges — the record itself is
never mutated with sync bookkeeping. `mix ash_multi_datalayer.inspect` grows
an outbox section (a query over the resource).

All transitions — enqueued, flushed, retried, rejected, conflicted, parked,
discarded, rebased, paused, resumed — emit telemetry under
`[:ash_multi_datalayer, :outbox, ...]` with queue-depth and oldest-entry-age
measurements, so sync lag is a dashboard number even before any UI exists.

## Conflict detection

Detection is what turns "someone else wrote the replica" from silent clobber
into a parked `:conflict` entry. Two modes:

- **`:off` (default): last-write-wins.** Flushes are blind PK-upserts of the
  full local record — identical semantics to ProvenCoverage propagation. This
  is sound *under the strategy's stated assumption* (below) and requires
  nothing from the resource.
- **`{:stale_check, field}` (opt-in):** updates/destroys flush as
  filter-guarded writes — `filter(pk == entry.pk and field == base_image.field)`
  — where `field` is a monotonic attribute the replica maintains
  (`:updated_at`, a `:lock_version` counter). Zero rows affected ⇒ conflict ⇒
  fetch the replica row into `remote_snapshot`, park as `:conflict`. Creates
  conflict when the PK/identity already exists remotely (the upsert reports a
  violation ⇒ same parking with the existing row).

**Multi-writer stance.** Without the notification bridge, the assumption is
a single writer per record — this node; `:off` silently clobbers other
writers and `{:stale_check, _}` detects-and-parks. **With** the bridge wired
(see Inbound changes), multiple clients are *supported in the convergence
sense*: your writes push through the outbox, other clients' writes pull in
via refresh, and a genuinely concurrent edit to the same record resolves as
last-write-wins under `:off` or a parked conflict under stale-check. What the
strategy still **never does is merge**: no CRDTs, no causal ordering, no
server reconciliation — notifications are at-most-once with gap-resync as
the backstop, and `rebase/2` remains the application's merge hook. The
sanctioned inbound writes are exactly `discard_local/1`, hydration, and
`refresh/3` (dirty-chain rule enforced on all of them).

## Kill switch

Per the behaviour ADR, kill-switch meaning is strategy-defined. For
LocalOutbox, engaging the switch means **synchronous write-through mode** —
the operator's lever when they've lost trust in async replication and want
divergence to stop growing:

- New writes: drain the record's pending PK chain inline, then write the
  replica synchronously *first*, then the local layer; a replica failure fails
  the operation (nothing committed anywhere). Authority is temporarily the
  replica's — exactly the conservative inversion an emergency demands.
- The queue keeps draining backlog; parked entries stay parked (they need
  resolution, not a mode switch).
- Reads stay local (there is nothing else complete to read from).

The kill switch and `pause_sync` are opposites, not variants: pause stops
replication and lets divergence grow (trusting the outbox to catch up);
the kill switch stops divergence and makes writes synchronous (distrusting
the outbox). `mix ash_multi_datalayer.disable/enable` work unchanged.

## What ships in v1 of the strategy

- `AshMultiDatalayer.Sync.OutboxEntry` extension (attributes, actions,
  trigger, notifier) + `mix ash_multi_datalayer.gen.outbox` igniter
  generator; oban/ash_oban/ash_sqlite as optional deps.
- Write path with transactional co-commit when local layer + outbox share a
  repo; per-PK chain-head flush; immediate MDL-owned job insert into the
  configured `oban_instance` + MDL sweep worker in its cron (see the
  Oban-instance note).
- `await/2`, `status/1`, `pending/2`, `parked/2`, resolution verbs
  (`retry/discard/discard_local/force/rebase`), `pause_sync`/`resume_sync`/
  `sync_paused?`, `refresh/3` with the dirty-chain rule, the
  `handle_external_change/2` + `handle_external_gap/2` inbound callbacks,
  outbox_ref metadata, full telemetry surface.
- Per-action layer targeting: side-effect-free `read_from:` context reads
  (shell-level, both strategies) and `write_through: true` context writes
  (LocalOutbox: single-action kill-switch semantics; ProvenCoverage: no-op).
- `conflict_detection: :off | {:stale_check, field}`.
- Full hydration (`:if_empty` / `:on_start` / `:manual`) with the
  outbox-not-empty guard.
- Verifiers: read_order/write_order shape, outbox resource contract, deps
  present, stale-check field exists and is trackable, single-node ack
  (unchanged — the multi-node story is the same as ProvenCoverage's), and
  rejection of multitenant resources whose local layer cannot isolate
  tenants (ash_sqlite answers `can?(:multitenancy) == false`) — a
  verify-time message, not a puzzled runtime `false`.

Deferred (each needs its own decision when real usage demands it): batched
flushes, cross-record transactional flush, entry-retention/audit mode
(entries are deleted on success in v1), any inbound sync beyond
`discard_local`, automated conflict-resolution strategies beyond the app's
own hooks.

## Open questions

1. **Upsert base images.** Local upserts have no reliable before-image (the
   same reason ProvenCoverage's invalidation does `drop_all` on upsert). With
   `{:stale_check, _}` on, upsert entries currently degrade to `:force`
   semantics — acceptable, or should upserts be rejected under stale-check?
2. **`await/2` across targets.** With multiple replication targets, is
   `:synced` all-targets (proposed) or per-target? All-targets is the safe
   default; per-target granularity can ride on `status/1` later.
3. **Entry payload encoding.** `payload`/`base_image` must round-trip through
   the outbox resource's data layer. SQLite: store as the resource's dumped
   form (`Ash.dump`-style casts) in a `:map`, rebuilt via `Ash.load`-free
   struct rebuild — the exact encoding is a **Phase-2 spike** deliverable
   (plan Phase 2 item 10; embedded types and `:term` fallback both
   plausible).
4. **`seq` generation.** The non-SQL variant is moot in v1 (outbox is
   SQL-only), but the SQLite variant is live (plan-review F7): only an
   `INTEGER PRIMARY KEY` autoincrements in SQLite, so with a uuid `id` PK,
   `seq` needs a source — integer PK/rowid (uuid demoted to identity) vs
   `max(seq)+1` inside the enqueue transaction. Plan Phase 2 item 9
   decides.
5. **Oban instance discovery.** `validate_opts/2` needs to know which Oban
   instance serves the queue (multi-instance apps): an `oban_instance` opt
   defaulting to `Oban`?
6. **Refresh scope for filtered refreshes.** `refresh(resource, filter)` with
   a filter needs deletion reconciliation *within the filter's scope only* —
   rows matching locally but absent remotely. Sound as long as the filter is
   evaluated on both sides; confirm the ash_remote wire encoder covers the
   needed predicates or restrict v1 to `pk | :all`.
7. **ProvenCoverage's bridge reaction.** With `handle_external_change/2` in
   the behaviour, ProvenCoverage *could* refresh-in-place (keep coverage
   warm) instead of invalidating. Tempting but subtle under at-most-once
   delivery (a missed notification breaks the "region still complete" claim
   that only the gap-drop currently repairs). v1 keeps invalidation;
   refresh-instead-of-invalidate is a separate proposal if wanted.

## Validation

- The **flagship mixed-strategy example** (grown from ash_remote_cache's
  authenticated collaborative-todo example — see the plan's example phase):
  Todo/TodoList on ash_sqlite + ash_remote under LocalOutbox with Oban Lite
  on the same SQLite file. All reads 0-RPC (counting router); writes commit
  locally with entries co-committed; offline toggle pauses the queue, writes
  accumulate, resume drains; a collaborator's edit under
  `{:stale_check, :updated_at}` parks a conflict that the **full
  conflict-resolution UI** resolves via every verb (three-way diff,
  field-level merge → `rebase/2`).
- Deterministic flush tests using the failing/blocking wrapper layer (the C3
  repro harness generalises): retry schedule (Oban `:test` manual mode),
  parking on exhausted transients vs immediate parking on rejection,
  chain-blocking + chain-head check under interleaving, `discard_local`,
  `rebase/2` re-entry, kill-switch inversion, pause/resume.
- Crash/restart test: kill between local commit and the job insert — sweeper
  recovers the entry (rows are the truth); co-commit case shows no window at
  all; ETS-local case recovers to replica state via hydration.
- Two-client scenario (bridge validation): client A's write arrives at
  client B as a notification → B's local layer refreshed, 1 RPC (the
  read-through), reads stay 0-RPC; B has a dirty pending write on the same
  record → refresh skips it and the flush parks a stale-check conflict; a
  forced websocket gap → `refresh(:all)` reconcile including a
  deleted-on-server row disappearing locally.
